Introduce-a-Bug Challenge

You've been finding and fixing other people's bugs. Now it's your turn to introduce bugs deliberately — then swap with a partner to fix. This is how real programmers do code review.

1
Read & Run
★★

Copy this working program into Thonny and run it to confirm it works. Read each line carefully — what does each one do?

celsius = int(input("Enter temp in °C: ")) fahrenheit = celsius * 9 / 5 + 32 print("In Fahrenheit: " + str(fahrenheit))
2
Introduce 3 Bugs
★★★

Introduce exactly 3 bugs into the temperature program. Use at least two different error types (e.g. missing colon, missing quote, misspelled variable). Keep a mental note of what you changed.

Do not tell your partner what you broke.

3
Swap & Fix
★★★

Swap your broken version with a partner. Before running it, try to spot the bugs by reading. Can you name the error type for each one?

Then run it in Thonny and fix all the bugs.

4
Silent Bug
★★★★

Can you introduce a bug that gives a wrong answer but no error message? These are the hardest bugs to find.

Hint: try changing a number or an operator in the formula. The program still runs — but the output is wrong.

Example: change * 9 / 5 + 32 to * 9 / 5 - 32. The program runs without errors, but every answer is wrong. Your partner needs to check the logic, not just the syntax.
5
Write Your Own
★★★★★

Write a brand new program from scratch. It must use input(), at least one if / else, and print different messages. Then break it in 2 places and swap again.

Idea: ask a user for their favourite colour, then print a message saying whether it matches yours. Use input() to get the colour and if colour == "blue": to decide the message.
💡 Real-world link: Professional programmers review each other's code every day — it's called code review. Being able to read unfamiliar code and spot bugs is one of the most valued skills in the industry.
